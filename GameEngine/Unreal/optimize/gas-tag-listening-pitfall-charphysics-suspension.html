GAS Tag 监听的隐形坑：声明的 Tag ≠ 运行时存在的 Tag

GAS Tag 监听的隐形坑：声明的 Tag ≠ 运行时存在的 Tag

继前一篇 Significance 预算与布娃娃 / PhysicsControl 的冲突修复 之后，我们在角色基类上加了一层「离屏即 suspend PhysicsAsset / Collision」的优化（ApplyPhysicsVisibilitySuspension），通过缓存「是否处于战斗」状态把战斗中的敌人豁免，让 weapon trace / AOE 仍能命中离屏目标。

但是上线后立刻撞到三个回归现象——根因极其朴素：监听的 Tag 名拼错了。本篇记录怎么定位、怎么修，并把一次性的诊断日志重构成可长期保留的 debug 通道。

1. 回归现象

#现象触发条件
a直接对战斗中敌人触发 knockout → 敌人飞天不论是否在渲染范围
b战斗中敌人离屏期间被 knockout → 拉回屏幕后飞天离屏 → 击中 → 回屏
c战斗中近距离打击 → 打不中敌人weapon trace 直接穿过敌人

2. 根因

2.1 Tag 名写错（核心）

监听器注册的 Tag 字符串是 OwnedTagsCategory.State.InCombatState，但 GAS Debugger（showdebug abilitysystem）显示敌人 ASC 上实际只有：

OwnedTagsCategory.State.InCombat        [2]
OwnedTagsCategory.State.InCombat.Attack [1]

没有 InCombatState。

InCombatState 在配置 ini 里是声明了的，但运行时没有任何系统给敌人加。仅有少数旧引用（StaticMeshDitherSubsystem、伴生系统）在 player 路径上使用——这也解释了为何 player 的 callback 能 fire 而敌人的不行。

后果：

  • 监听 InCombatState 永远不 fire
  • HasMatchingGameplayTag(InCombatState) 永远返回 false
  • bIsInCombatCached 在 enemy 上永远为 false
  • ApplyPhysicsVisibilitySuspension 的 bExempt = IsPlayer || bIsInCombatCached || IsRagdollActive() || bForceEnablePhysicsControl 不豁免战斗中敌人
    • 离屏即被 suspend：SetPhysicsAsset(nullptr) + SetCollisionEnabled(NoCollision)
    • 现象 c：weapon trace 穿过去（无 Collision body）
    • 现象 b：离屏 suspend 后再触发 knockout，PhysicsControl 找不到 body 路径，回屏时强制重建 ragdoll 状态不一致 → 升天

2.2 直接 knockout 也飞天（现象 a）

IsRagdollActive() 此前被改成基于 bIsInCombatCached 的简化判断，导致非战斗豁免链断裂。已恢复为原生检查：

LocalMesh->IsSimulatingPhysics()
|| 任一 CachedPhysicsControlComponent->PrimaryComponentTick.IsTickFunctionEnabled()

这样在 knockout 触发瞬间，PhysicsControl tick 一上线立刻进入 IsRagdollActive()=true，IfCanSkipTick() 短路返回 false，Budget 不再 skip tick。

3. 诊断过程（值得复盘）

步骤 1：缩小路径

现象 a 与 c 在不同代码路径。先假设两套逻辑都靠 bIsInCombatCached，注释掉 ApplyPhysicsVisibilitySuspension 后 c 缓解 → 验证了 suspension 路径是 c 的元凶。

步骤 2：确认 listener 是否 fire

在 OnInCombatStateTagChanged 入口加 log → player 触发，敌人永远不触发。

步骤 3：定位「为啥不 fire」

最初怀疑两个 ASC 取法（GetAbilitySystemComponent() 与 GetComponentByClass<UAbilitySystemComponent>()）在 modular character 下指向不同 ASC。在 Tick() 加每秒一次的双 ASC 指针对比日志：

[InCombatDiag] BP_Infected_C_0 ASC_A=0x... (count=0) ASC_B=0x... (count=0) same=1

same=1 排除 ASC 实例分歧，count=0 暴露真相：两边的 ASC 都说没这个 tag。结合 GAS Debugger 截图上的 InCombat [2]，结论唯一：监听的 Tag 字符串就是错的。

步骤 4：grep 验证

搜 OwnedTagsCategory.State.InCombat 的真实用法 → ini 子 Tag 全是 InCombat.*（Attack / ComboSelected / Disengaging / HasTarget / Move.Chase 等），父 Tag InCombat 由子 Tag 隐式 propagate 出 count。InCombatState 是孤儿 Tag。

4. 修复

4.1 Tag 名校正

4 处 OwnedTagsCategory.State.InCombatState → OwnedTagsCategory.State.InCombat：

  • BeginPlay 监听注册
  • EndPlay 解注册
  • FreezeDeathRagdoll 死亡清理 RemoveAllLooseGameplayTag
  • Tick 内的诊断 var

监听器仍用 EGameplayTagEventType::AnyCountChange：父 Tag InCombat 的 count 会随子 Tag（InCombat.Attack 等）增减而变化，恰好等价于「是否处于任何战斗子状态」。

4.2 恢复 ApplyPhysicsVisibilitySuspension

之前为隔离问题改成「无条件 restore + early return」的临时屏蔽块已移除，函数恢复原有 exempt 判定逻辑。bIsInCombatCached 现在为敌人正确返回 true，战斗中敌人即使离屏也不会被 strip。

4.3 IsRagdollActive() 恢复原生检查

参见 §2.2，把「读 bIsInCombatCached」的简化版回退为查 IsSimulatingPhysics() + PhysicsControl tick enabled。

5. 调试通道沉淀（长期保留）

为了下次类似问题不再「加 LogTemp Warning → 跑测试 → 删 log」循环，把这次的诊断日志重构成可持续保留的 debug hook。

5.1 新增专用日志分类

// 默认 Verbose（被过滤），需要时显式开启
GAMEMODULE_API DECLARE_LOG_CATEGORY_EXTERN(LogCharPhysics, Verbose, All);

5.2 两类日志点

位置内容触发频率
BeginPlay 监听注册后ASC 指针 / 初始 tag count / 初始 cached / IsPlayer每次角色生成一次
OnInCombatTagChanged 入口NewCount / IsPlayer每次 InCombat 父 tag 计数变化
Tick 内每帧 ragdoll dumpbucket / simRoot / simAny / PCtick / skipTick / forcePC / ragActive / exempt / offScrSec / pelvisZ二级门控，手动开启

5.3 二级门控

每帧 Tick log 的数据采集（socket 查找、CachedPhysicsControlComponents 遍历、TimeSince 计算）成本不可忽略。采用：

static TAutoConsoleVariable<int32> CVarCharPhysicsTick(
    TEXT("Game.Debug.CharPhysicsTick"), 0,
    TEXT("Enable per-frame char physics dump"));

if (CVarCharPhysicsTick.GetValueOnGameThread() > 0
    && !IsPlayer && bIsInCombatCached
    && UE_LOG_ACTIVE(LogCharPhysics, Verbose))
{
    // ... 重数据采集 + UE_LOG ...
}

三层短路：

  • CVar Game.Debug.CharPhysicsTick：runtime 主开关，关闭时数据采集完全短路。
  • UE_LOG_ACTIVE(LogCharPhysics, Verbose)：即便 CVar 开了，如果分类未提级，数据采集也跳过。
  • 类别本身默认 Verbose：所有 LogCharPhysics 默认就被 LogSuppress 过滤掉，0 输出 0 干扰。

5.4 使用速查

# 关键事件流（生成 / Tag 切换）
log LogCharPhysics Verbose

# 怀疑离屏 knockout 起飞，加每帧 dump
log LogCharPhysics Verbose
Game.Debug.CharPhysicsTick 1

# 持久化（DefaultEngine.ini）
[Core.Log]
LogCharPhysics=Verbose

7. 经验

  1. GAS Tag 用前一定要 showdebug abilitysystem 对照。ini 里 declared 的 Tag 不等于运行时真的有人加。声明 Tag 不会触发任何编译错误，完全是运行时静默失败。
  2. 修 GAS 监听 bug 时，第一条要加的日志是「callback 入口」而不是「业务分支」。能直接证伪「listener 不 fire」假设，避免一路往下查冤枉路。
  3. 诊断日志值得沉淀成专用 log category，而不是 LogTemp Warning。下次类似问题能直接 log LogXxx Verbose 拿到现成的事件流，不用重新埋点。
  4. 数据采集和打印分两层门控：分类决定「打不打」，CVar 决定「算不算」。重数据 + 高频路径必备这个结构。
  5. 父 Tag 计数 ≠ 子 Tag 存在：在 GAS 里监听父 Tag 的 AnyCountChange 比监听具体子 Tag 灵活——业务上「是否处于任何战斗子状态」恰好等价于父 Tag count > 0。
